# Deploy Smart Compta — Neon (Postgres) + Render (backend) + Vercel (frontend)

100% free, **persistent** Postgres (Neon doesn't expire). ~15 minutes.

## 0. Prerequisites
- A **GitHub** account (push this repo there).
- A **Neon** account (neon.tech), a **Render** account (render.com), and a
  **Vercel** account (vercel.com) — all sign in with GitHub.
- A **public** Make/Integromat webhook URL (NOT a `.local` address — cloud hosts
  cannot reach `.local`). E.g. `https://hook.we.make.com/xxxxxxxx`.

## 1a. Database → Neon (do this first)
1. neon.tech → **New Project** (pick a region near you).
2. Copy the **connection string** it shows, e.g.
   `postgresql://user:pass@ep-xxxx.aws.neon.tech/neondb?sslmode=require`
3. Keep it handy — you'll paste it as `DATABASE_URL` in Render (step 2).

## 1b. Push the repo to GitHub
```bash
cd /Users/d.boulmechaour/Desktop/acount
git init
git add .
git commit -m "Smart Compta"
git branch -M main
git remote add origin https://github.com/<you>/smart-compta.git
git push -u origin main
```

## 2. Backend → Render (uses render.yaml)
1. Render dashboard → **New +** → **Blueprint**.
2. Pick your GitHub repo. Render reads `render.yaml` and proposes
   `smartcompta-api` (Python web service, root `backend/`).
3. It will ask for the env vars marked `sync: false`. Set at least:
   - `DATABASE_URL` = your **Neon** connection string (from step 1a)
   - `WEBHOOK_URL` = your **public** webhook URL
   (you can fill `CORS_ALLOWED_ORIGINS` after the frontend is up.)
4. Click **Apply**. The build runs `collectstatic`, `migrate`, and `seed`
   automatically against Neon (so the demo accounts exist in your Neon DB).
5. When live, note the URL, e.g. `https://smartcompta-api.onrender.com`.
6. Back in **smartcompta-api → Environment**, set:
   - `CORS_ALLOWED_ORIGINS` = your Vercel URL (fill after step 3, e.g.
     `https://smart-compta.vercel.app`)
   - `ALLOWED_HOSTS` can stay empty (Render host is auto-added).
   Save → it redeploys.

> Free Render services **sleep after ~15 min idle**; the first request then
> takes ~30–60s to wake. Normal for the free tier.

## 3. Frontend → Vercel
1. Vercel → **Add New… → Project** → import the same GitHub repo.
2. **Root Directory:** set to **`frontend`** (important — the repo is a monorepo).
3. Framework preset: **Next.js** (auto-detected). Leave build/output defaults.
4. **Environment Variables:**
   - `NEXT_PUBLIC_API_URL` = your Render backend URL (from step 2.4)
   - `WEBHOOK_URL` = your public webhook URL (server-side; used by the
     `/api/webhook` bridge route)
   - `NEXT_PUBLIC_APP_NAME` = `Smart Compta`
5. **Deploy.** You get e.g. `https://smart-compta.vercel.app`.
6. Go back to Render and set `CORS_ALLOWED_ORIGINS` to this exact Vercel URL
   (no trailing slash), then let it redeploy.

## 4. Verify
- Open the Vercel URL → log in with `comptable` / `comptable`.
- If login hangs on the first try, the Render backend is waking from sleep —
  retry after ~30s.

## 5. Updating
Push to `main` → both Render and Vercel auto-redeploy.

## Environment variables reference

### Render (backend)
| Key | Value |
|-----|-------|
| `DATABASE_URL` | your Neon connection string (`postgresql://…sslmode=require`) |
| `DJANGO_SECRET_KEY` | auto-generated by Render |
| `DJANGO_DEBUG` | `False` |
| `WEBHOOK_URL` | your public Make/Integromat webhook |
| `CORS_ALLOWED_ORIGINS` | your Vercel URL |
| `ALLOWED_HOSTS` | optional (Render host auto-added) |
| `CLOUDINARY_URL` | `cloudinary://<key>:<secret>@<cloud_name>` (persistent image storage) |

### Vercel (frontend)
| Key | Value |
|-----|-------|
| `NEXT_PUBLIC_API_URL` | your Render backend URL |
| `WEBHOOK_URL` | your public webhook (server-side only) |
| `NEXT_PUBLIC_APP_NAME` | `Smart Compta` |

## Notes & gotchas
- **`.local` webhooks won't work in the cloud.** Use a public URL in prod.
- **Scanner** needs the Make/Integromat scenario to return the extraction JSON
  via a **Webhook Response** module (otherwise it replies `Accepted` and the
  app shows the error — by design).
- `seed` runs on every Render build, so demo accounts always exist. It is
  idempotent (uses `get_or_create`), so it won't duplicate data.
- To stop the public demo accounts being created in prod, remove `&& python
  manage.py seed` from `buildCommand` in `render.yaml`.
